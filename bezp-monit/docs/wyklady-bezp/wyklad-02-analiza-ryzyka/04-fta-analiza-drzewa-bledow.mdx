# FTA - analiza drzewa błędów

<LearningObjective>
Po tej sekcji student potrafi zbudować drzewo błędów (Fault Tree Analysis), zrozumie symbole i logikę FTA, obliczy prawdopodobieństwo zdarzeń top-level oraz zastosuje FTA do analizy krytycznych scenariuszy w instalacjach OZE.
</LearningObjective>

## Wprowadzenie do Fault Tree Analysis

### Definicja i cel FTA

**Fault Tree Analysis (FTA)** to **dedukcyjna metoda analizy ryzyka**, która analizuje **jak konkretne zdarzenie niepożądane** (top event) może wystąpić poprzez **kombinację bardziej podstawowych awarii**.

**Kluczowe założenia FTA**:
- **Top-down approach** - od konsekwencji do przyczyn
- **Systematyczna analiza** wszystkich możliwych ścieżek
- **Ilościowa ocena** prawdopodobieństwa zdarzeń
- **Identyfikacja** najważniejszych czynników ryzyka

<KeyConcept title="Podstawowe elementy FTA">
**Top Event** - główne zdarzenie niepożądane, które analizujemy

**Basic Events** - podstawowe awarie, które nie są dalej rozwijane

**Intermediate Events** - zdarzenia pośrednie, które mogą być rozłożone na składniki

**Gates** - operatory logiczne (AND, OR, NOT) łączące zdarzenia
</KeyConcept>

### Zastosowania FTA w OZE

**Typowe top events w instalacjach OZE**:

**Dla farm PV**:
- Pożar farmy PV
- Porażenie prądem podczas serwisu
- Awaria systemu monitoringu
- Utrata stabilności sieci

**Dla turbin wiatrowych**:
- Pęknięcie łopatki turbiny
- Pożar gondoli
- Utrata kontroli nad turbiną
- Ciężkie obrażenia podczas serwisu

**Dla BESS**:
- Thermal runaway w magazynie energii
- Eksplozja/pożar instalacji BESS
- Uwolnienie toksycznych gazów
- Utrata funkcji systemu bezpieczeństwa

**Dla biogazowni**:
- Eksplozja w strefie ATEX
- Uwolnienie toksycznego gazu (H2S)
- Pożar instalacji biogazowej
- Zanieczyszczenie środowiska

## Symbole i logika FTA

### Podstawowe symbole FTA

**1. Gate symbols (Symbole bramek logicznych)**

```
AND Gate (∧) - Wszystkie zdarzenia muszą wystąpić jednocześnie
OR Gate (∨) - Przynajmniej jedno zdarzenie musi wystąpić
NOT Gate (¬) - Zdarzenie A nie występuje
```

**2. Event symbols (Symbole zdarzeń)**

```
Basic Event (○) - Podstawowa awaria, nie rozwijana dalej
Intermediate Event (□) - Zdarzenie pośrednie, rozwinięte dalej
Top Event (○) - Główne zdarzenie niepożądane
Undeveloped Event (◇) - Zdarzenie nie rozwinięte (brak danych)
```

**3. Transfer symbols (Symbole transferu)**

```
Transfer IN (▶) - Wprowadzenie z innej części drzewa
Transfer OUT (◀) - Przekazanie do innej części drzewa
```

### Logika bramek

**Bramka AND (I)**:
- **Warunek**: Wszystkie zdarzenia wejściowe muszą wystąpić
- **Prawdopodobieństwo**: P(A AND B) = P(A) × P(B)
- **Przykład**: Pożar = (Źródło ognia) AND (Materiał palny) AND (Utleniacz)

**Bramka OR (LUB)**:
- **Warunek**: Co najmniej jedno zdarzenie wejściowe musi wystąpić  
- **Prawdopodobieństwo**: P(A OR B) = P(A) + P(B) - P(A AND B)
- **Uproszczenie**: Jeśli P(A), P(B) `<< 1`, to P(A OR B) ≈ P(A) + P(B)
- **Przykład**: Awaria inwertera = (Awaria elektroniki) OR (Awaria chłodzenia) OR (Błąd oprogramowania)

<Example title="FTA dla top event: Pożar inwertera PV">
**Top Event**: Pożar inwertera PV 500 kW

**Struktura drzewa**:
```
                    Pożar inwertera
                        |
                   ┌─────────┐
                   │   OR    │
             ┌─────────┬─────────┐
             │         │         │
        Przegrzanie   Zwarcie   Błąd kontroli
             │         │         │
        ┌────────┐ ┌──────┐  ┌──────────┐
        │  AND   │ │ OR   │  │    OR    │
        └────────┘ └──────┘  └──────────┘
```

**Rozwinięcie**:

**Przegrzanie = (Wysoka temperatura) AND (Brak gaszenia)**
- Wysoka temperatura: P = 1×10⁻⁴ /rok (awarie chłodzenia)
- Brak gaszenia: P = 5×10⁻² (brak systemu detekcji/gaszenia)
- P(Przegrzanie) = 1×10⁻⁴ × 5×10⁻² = 5×10⁻⁶ /rok

**Zwarcie = (Uszkodzenie izolacji) OR (Przepięcie)**
- Uszkodzenie izolacji: P = 2×10⁻³ /rok
- Przepięcie: P = 1×10⁻⁴ /rok  
- P(Zwarcie) ≈ 2×10⁻³ /rok

**Błąd kontroli = (Software bug) OR (Hardware failure)**
- Software bug: P = 5×10⁻³ /rok
- Hardware failure: P = 1×10⁻³ /rok
- P(Błąd kontroli) ≈ 6×10⁻³ /rok

**Całkowite prawdopodobieństwo pożaru**:
P(Pożar) = P(Przegrzanie) + P(Zwarcie) + P(Błąd kontroli)
P(Pożar) = 5×10⁻⁶ + 2×10⁻³ + 6×10⁻³ = 8×10⁻³ /rok

**Interpretacja**: Ryzyko pożaru inwertera raz na 125 lat - wymaga działań redukcyjnych.
</Example>

## Proces budowania FTA

### Krok 1: Definicja top event

**Kryteria dobrego top event**:
- **Konkretny** - jasno zdefiniowany, mierzalny
- **Znaczący** - ma realne konsekwencje dla bezpieczeństwa
- **Obserwowalny** - można stwierdzić czy wystąpił
- **Granice systemu** - jasne co wchodzi w zakres analizy

**Przykład poprawnej definicji**:
❌ **Złe**: "Awaria systemu bezpieczeństwa"
✅ **Dobre**: "Uwolnienie palnego gazu CH4 > 100 ppm w pomieszczeniu technicznym"

### Krok 2: Identyfikacja przyczyn bezpośrednich

**Metody identyfikacji**:
- **"5 Why" analysis** - pytania dlaczego
- **Brainstorming** zespołu ekspertów
- **Analiza dokumentacji** - raporty awarii, FMEA
- **Expert judgment** - wiedza ekspertów branżowych

### Krok 3: Rozwinięcie drzewa

**Zasady budowania drzewa**:
- **Każdy poziom** analizuje "jak" zdarzenie może wystąpić
- **Identyfikacja** wszystkich znaczących przyczyn
- **Zatrzymanie** na basic events (nie da się dalej rozłożyć)
- **Sprawdzenie** kompletności i logiki

### Krok 4: Analiza ilościowa

**Dane wymagane**:
- **Prawdopodobieństwa** basic events (z danych historycznych, ekspertów)
- **Czasowe zależności** między zdarzeniami (jeśli występują)
- **Zależności** między zdarzeniami (common cause failures)

**Obliczenia matematyczne**:
- **Propagacja** prawdopodobieństw od basic events do top event
- **Minimal cut sets** - minimalne kombinacje awarii prowadzące do top event
- **Sensitivity analysis** - które zdarzenia najbardziej wpływają na ryzyko

## Minimal Cut Sets i analiza krytyczności

### Minimal Cut Sets

**Definicja**: Minimal Cut Set to **minimalna kombinacja basic events**, której wystąpienie powoduje realizację top event.

**Znaczenie**:
- **Identyfikacja** najkrótszych ścieżek awarii
- **Priorytetyzacja** środków redukcji ryzyka
- **Analiza** skuteczności środków ochronnych

<Example title="Minimal Cut Sets dla pożaru inwertera PV">
**Z wcześniejszego przykładu**, rozwijając drzewo bardziej szczegółowo:

**Top Event**: Pożar inwertera PV

**Minimal Cut Sets (1st order)** - pojedyncze awarie:
1. [Awaria chłodzenia] - P = 1×10⁻⁴ /rok
2. [Zwarcie w elektronice] - P = 2×10⁻³ /rok  
3. [Błąd programowy] - P = 5×10⁻³ /rok

**Minimal Cut Sets (2nd order)** - kombinacje dwóch awarii:
4. [Błąd detekcji] AND [Brak gaszenia] - P = 1×10⁻⁵ × 5×10⁻² = 5×10⁻⁷ /rok
5. [Przeciążenie] AND [Brak zabezpieczeń] - P = 2×10⁻⁴ × 1×10⁻² = 2×10⁻⁶ /rok

**Analiza ważności**:
- **Najbardziej krytyczne**: Błąd programowy (P = 5×10⁻³)
- **Drugie w kolejności**: Zwarcie w elektronice (P = 2×10⁻³)
- **Trzecie**: Awaria chłodzenia (P = 1×10⁻⁴)

**Zalecenia**:
1. **Priorytet 1**: Poprawa jakości oprogramowania, redundancja
2. **Priorytet 2**: Dodatkowe zabezpieczenia elektryczne
3. **Priorytet 3**: Redundantne systemy chłodzenia
</Example>

### Analiza wrażliwości (Sensitivity Analysis)

**Cel**: Określenie, które basic events najbardziej wpływają na prawdopodobieństwo top event.

**Metody**:
- **Zmiana** prawdopodobieństwa pojedynczego basic event o 10%
- **Obserwacja** wpływu na prawdopodobieństwo top event
- **Ranking** basic events według wpływ

**Aplikacja**: Pomaga w **alokacji zasobów** - skupiamy się na basic events o największym wpływie.

## Praktyczne zastosowania FTA w OZE

### FTA dla systemów bezpieczeństwa BESS

**Typowy top event**: "Thermal runaway w module BESS"

**Struktura analizy**:
```
Thermal Runaway = (Pojedyncza awaria ogniwa) AND (Brak wykrycia) AND (Brak interwencji)
```

**Rozwijanie każdego komponentu**:
- **Pojedyncza awaria ogniwa** = (Defekt produkcyjny) OR (Nadmierne ładowanie) OR (Uszkodzenie mechaniczne)
- **Brak wykrycia** = (Awaria czujnika temperatury) OR (Błąd BMS) OR (Zaniedbanie monitoringu)
- **Brak interwencji** = (Awaria systemu gaszenia) OR (Błąd alarmu) OR (Brak personelu)

### FTA dla turbin wiatrowych

**Top event**: "Pęknięcie łopatki turbiny wiatrowej"

**Kluczowe czynniki**:
- **Czynniki projektowe**: Materiał, geometria, obliczenia wytrzymałościowe
- **Czynniki środowiskowe**: Ekstremalne wiatry, lód, erozja deszczem
- **Czynniki operacyjne**: Vibracje, cykle zmęczeniowe, uderzenia piorunów
- **Czynniki utrzymania**: Inspekcje, naprawy, monitoring stanu

<KeyConcept title="Specyfika FTA dla OZE">
**Długi okres eksploatacji** - 20-30 lat wymaga uwzględnienia degradacji w czasie

**Warunki środowiskowe** - ekstremalne zjawiska pogodowe jako basic events

**Nowe technologie** - ograniczone dane historyczne wymagają konserwatywnych oszacowań

**Cybernetyki** - software bugs, hacking jako nowe kategorie basic events
</KeyConcept>

### Integracja FTA z innymi metodami

**FTA + FMEA**:
- **FMEA** identyfikuje failure modes komponentów
- **FTA** analizuje jak failure modes łączą się w top event

**FTA + HAZOP**:
- **HAZOP** identyfikuje odchylenia procesowe
- **FTA** modeluje jak odchylenia prowadzą do krytycznych konsekwencji

**FTA + QRA**:
- **FTA** dostarcza modele probabilistyczne
- **QRA** używa wyników FTA do oszacowania ryzyka finansowego

## Narzędzia i software dla FTA

### Ręczne obliczenia vs. software

**Ręczne obliczenia** - odpowiednie dla:
- **Prostych drzew** (do 20-30 basic events)
- **Wstępnej analizy** koncepcyjnej
- **Zrozumienia** metody i logiki

**Software** - wymagany dla:
- **Złożonych drzew** (>50 basic events)
- **Precyzyjnych obliczeń** prawdopodobieństw
- **Analizy minimal cut sets**

### Popularne narzędzia

**Komercjalne**:
- **ReliaSoft RENO** - zaawansowane narzędzia do analizy niezawodności
- **Isograph FaultTree+** - specjalistyczny software do FTA
- **ASENT Reliability** - kompletny pakiet do analizy ryzyka

**Open source**:
- **OpenFTA** - darmowy software do analizy FTA
- **Python libraries** - numpy, scipy do obliczeń matematycznych
- **R language** - zaawansowane analizy statystyczne

<Example title="Implementacja FTA dla farmy PV 50 MW - wyniki i rekomendacje">
**Kontekst**: Farma PV 50 MW, analiza 3 krytycznych top events

**Top Event 1**: Pożar farmy PV (uszkodzenie >1 MW)
- **Minimal Cut Sets**: 12 kombinacji awarii
- **Prawdopodobieństwo**: 2.3×10⁻³ /rok (raz na 435 lat)
- **Główne czynniki**: Łuk elektryczny (60%), uderzenie pioruna (25%)

**Top Event 2**: Porażenie prądem podczas serwisu
- **Minimal Cut Sets**: 8 kombinacji awarii  
- **Prawdopodobieństwo**: 1.1×10⁻⁴ /rok (raz na 9100 lat)
- **Główne czynniki**: Brak LOTO (70%), błąd procedury (20%)

**Top Event 3**: Awaria systemu monitoringu
- **Minimal Cut Sets**: 15 kombinacji awarii
- **Prawdopodobieństwo**: 8.7×10⁻² /rok (raz na 11 lat)
- **Główne czynniki**: Cyberatak (40%), awarie sprzętu (35%)

**Rekomendacje na podstawie FTA**:
1. **Zabezpieczenia AFCI** na poziomie stringów (redukcja ryzyka o 80%)
2. **Systemy LOTO** z automatyczną kontrolą (redukcja o 90%)
3. **Cyberbezpieczeństwo** - segmentacja, firewall, monitoring (redukcja o 70%)

**ROI analizy FTA**: Koszt €120k, oszczędności przez uniknięcie awarii €2.1M - zwrot inwestycji 1750%.
</Example>

<InstructorNotes title="📝 Notatki prowadzącego">
Podaj intuicję: FTA (Fault Tree Analysis) jest dedukcyjne — zaczyna od zdarzenia szczytowego i rozkłada je na kombinacje przyczyn, używając bramek AND/OR. To przeciwieństwo FMEA, które idzie „od dołu” (od failure modes do skutków).

Wyjaśnij podstawowe symbole i zasady niezależności zdarzeń wejściowych. Na prostym przykładzie pożaru inwertera PV pokaż, jak obliczenia dla bramek AND i OR prowadzą do oszacowania prawdopodobieństwa top event oraz jak wyznaczać minimal cut sets (minimalne kombinacje awarii prowadzące do zdarzenia).

Podkreśl praktykę: najpierw zdefiniuj dobre top event (konkretne, mierzalne, o jasnych granicach systemu), potem systematycznie rozwijaj drzewo i weryfikuj kompletność. Zwróć uwagę na common cause failures i zależności — uproszczenia są dopuszczalne, ale trzeba je zaznaczać.

Połącz FTA z innymi metodami: wyniki FMEA zasilają FTA (przyczyny), a FTA dostarcza modele do analiz ilościowych i priorytetyzacji działań. Dla OZE wskaż typowe top events: thermal runaway w BESS, pożary PV, pęknięcie łopatki, zdarzenia w strefach ATEX.
</InstructorNotes>
