# Filozofia alarmowania

## Poziomy priorytetów

### P1 - Krytyczny (Critical)
- **Definicja**: Zagrożenie bezpieczeństwa
- **Czas reakcji**: < 5 minut
- **Przykłady**: Pożar, thermal runaway, zwarcie DC

### P2 - Wysoki (High)
- **Definicja**: Poważna degradacja wydajności
- **Czas reakcji**: < 1 godzina
- **Przykłady**: Awaria inwertera, string fault

### P3 - Średni (Medium)
- **Definicja**: Spadek wydajności
- **Czas reakcji**: < 24 godziny
- **Przykłady**: PR < 80%, temperatura anomalia

### P4 - Niski (Low)
- **Definicja**: Informacyjne
- **Czas reakcji**: < 7 dni
- **Przykłady**: Maintenance reminder, firmware update

## Reguły alarmowe

### Przykład: Awaria inwertera
```yaml
Nazwa: INV_FAILURE
Warunek: (P_AC < 1 kW) AND (Irradiance > 300 W/m²)
Czas: 5 minut
Priorytet: P2
Akcja: SMS + email + dashboard highlight
```

### Przykład: String underperformance
```yaml
Nazwa: STRING_FAULT
Warunek: I_string < 0.8 × I_avg FOR 15min
Priorytet: P3
Akcja: Email serwis + dashboard marker
```

## Anti-flooding mechanisms

### Histereza
- **Zasada**: Różne progi dla alarmu ON/OFF
- **Przykład**: Alarm ON: T > 75°C, Alarm OFF: T < 70°C

### Dead-band
- **Zasada**: Strefa nieczułości dla danych analogowych
- **Przykład**: ±2% dla pomiarów mocy

### Suppression
- **Zasada**: Ukrywanie alarmów pochodnych
- **Przykład**: Jeśli "Inwerter OFF" → ukryj "String fault"
